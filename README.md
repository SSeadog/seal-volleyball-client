# SealVolleyball

2:2 실시간 대전 배구 게임. Colyseus 기반 WebSocket 멀티플레이를 통해 웹(Web) 환경에서 동기화된 경기를 플레이할 수 있습니다.

## 제작 정보
- 장르: 2:2 실시간 대전 배구
- 제작 인원: 1명
- 제작 기간: 약 2개월
- 사용 엔진: UnityEngine, Colyseus(JavaScript)
- 플랫폼: web
- 배포(itch.io): https://sseadog.itch.io/seal-volleyball

## 핵심 게임 로직

### 1. 실시간 멀티플레이 네트워크 아키텍처 설계
Colyseus 기반 WebSocket 멀티플레이 서버를 구축하고, `LobbyRoom` / `GameRoom`으로 멀티룸 아키텍처를 분리했습니다. 매칭 큐 기반 4인 자동 매칭 및 게임 룸 생성 로직을 구현했습니다.

### 2. Input 기반 네트워크 동기화 시스템 구현
클라이언트 입력을 서버로 전송하고, 서버에서 동일한 시뮬레이션 구조로 게임 상태를 계산하여 동기화하는 방식을 설계했습니다.

### 3. Tick 기반 게임 동기화 시스템 구현
서버와 클라이언트가 동일 Tick 기준으로 물리 시뮬레이션을 실행하도록 구조를 잡았습니다. 지연 시간 기반 위치 보간도 고려했지만, 프로젝트 규모를 고려하여 기본 Tick 동기화 방식 중심으로 구현했습니다.

### 4. 네트워크 지연(Latency) 및 Jitter 대응
네트워크 지연 허용 범위를 최대 3 Tick까지 적용했습니다. 서버는 동기화 안정성을 위해 2 Tick 지연된 과거를 기준으로 시뮬레이션하도록 구성했습니다. Jitter 대응은 우선 지연 허용 범위를(5 Tick) 늘리는 방식으로 해결했고, 추후 지연 시간 기반 위치 보간 방식 적용을 검토하고 있습니다.

### 5. 물리 기반 공 및 충돌 시뮬레이션 구현
서버 기반으로 공/플레이어/네트 충돌을 포함한 물리 시뮬레이션을 구현했습니다. Unity 물리 엔진 의존도를 낮추고, 서버와 클라이언트에서 동일한 계산 로직을 적용하는 방향으로 접근했습니다.

## 시행착오 / 문제점

### 웹 배포 과정에서의 어려움
- 싱글 스레드 기반 환경에서 await 연속 호출이 메인 스레드에서 안정적으로 처리되지 않는 이슈 발생했습니다.
- 이에 Task를 캐싱해두고 Update()에서 IsCompleted를 폴링하는 구조로 변경하였습니다.

### AI 활용 리소스 추출
- 리소스 자체 제작을 위해 시도해봤지만, 기대만큼 결과가 나오진 않았습니다.
- 특히 캐릭터는 잘 나오면 큰 의미가 있을 것 같았지만, 움직임이 이상하게 나오는 문제가 있었습니다(예: 애니메이션/모션의 부자연스러움).
- 다음 프로젝트에서는 더 연구하여 조금 더 좋은 리소스를 만들어보고 싶습니다.

## 향후 개선 방향
- 지연 시간 기반 위치 보간(Interpolation) 도입으로 Jitter 대응 품질을 향상
- Tick 동기화 안정성 및 동기화 오류 디버깅 도구 강화
- 서버 검증(입력/행동) 강화 및 예외 케이스 처리 확장
